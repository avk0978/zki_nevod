# ТЗ Ячейки (Cell)

## Определение

Ячейка — пользователь сети Nevod.
Идентичность ячейки — криптографическая, не привязана к конкретному узлу.
Ячейка может существовать на нескольких узлах одновременно (home + visiting).

---

## Идентификация ячейки

- **cell_id** — Ed25519 публичный ключ ячейки (master keypair)
- **cell_secret** — Ed25519 приватный ключ (хранится только локально)
- **display_name** — отображаемое имя (опционально)

Адрес ячейки в контексте узла: `cell_id@node_id`

---

## Несколько устройств (multi-device)

Одна ячейка может работать на нескольких устройствах одновременно.
Master keypair остаётся на первом (основном) устройстве.
Каждое дополнительное устройство имеет свой device keypair, авторизованный master-ключом.

### Структура
```
cell_id (master pubkey)
  ├── device_1 (телефон)   — device_key_1, подписан master
  ├── device_2 (ноутбук)   — device_key_2, подписан master
  └── device_3 (планшет)   — device_key_3, подписан master
```

### Добавление устройства
```
1. На новом устройстве генерируется device keypair
2. Основное устройство сканирует QR-код нового (или вводит pairing code)
3. Master-ключ подписывает device_key → device_cert
4. Home-узел получает обновлённый список авторизованных устройств
5. Сообщения доставляются на все активные устройства ячейки
```

### Отзыв устройства
```
1. С любого авторизованного устройства: команда "revoke device_N"
2. Master-ключ подписывает revocation
3. Home-узел обновляет список → device_N больше не получает сообщения
```

### Потеря основного устройства (с master-ключом)
- Восстановление невозможно (конфиденциальность важнее удобства)
- Ячейка создаётся заново с новым keypair
- Существующие контакты нужно уведомить о новом cell_id вручную

---

## Home-узел и visiting-узел

### Home-узел
- Постоянный узел где зарегистрирована ячейка
- Хранит presence_table для ячейки
- Буферизирует сообщения когда ячейка оффлайн
- Знает на каких эфемерных узлах сейчас находится ячейка

### Visiting-узел (эфемерный)
- Временный узел (например, для поездки)
- Ячейка подключается к нему через приглашение
- При уничтожении visiting-узла — переписка на нём исчезает
- Home-узел обновляет presence_table при подключении/отключении

---

## Жизненный цикл ячейки

### Регистрация
```
1. Пользователь генерирует keypair на устройстве
2. Выбирает узел для регистрации (home-узел)
3. Отправляет cell_id (pubkey) + display_name на узел
4. Проходит аутентификацию: ZKP + TOTP
5. Узел регистрирует ячейку, сохраняет pubkey
6. Ячейка зарегистрирована
```

### Подключение к visiting-узлу
```
1. Получает invite-токен от home-узла или другой ячейки
2. Предъявляет токен + проходит ZKP на новом узле
3. Новый узел уведомляет home-узел: "alice теперь и у меня"
4. Home-узел обновляет presence_table
```

### Отключение от visiting-узла
```
Явное: ячейка отправляет команду "leave" → узел уведомляет home
Неявное: visiting-узел уничтожен → home обновляет presence_table
```

## Отправка сообщения

```
1. Ячейка формирует сообщение
2. Получает pubkey адресата (из контактов)
3. Шифрует payload ключом адресата (X25519 + ChaCha20)
4. Подписывает сообщение своим Ed25519 ключом
5. Отправляет на свой текущий узел
6. Узел берёт на себя маршрутизацию
```

Узел не может расшифровать payload — только доставить.

---

## Структура контакта

```
contact:
  cell_id      — публичный ключ контакта
  display_name — имя (задаётся локально)
  home_node    — известный home-узел контакта
  last_seen    — последнее время онлайн
```

---

## Типы сообщений

| Тип | Описание | Ограничение |
|-----|----------|-------------|
| text | Текстовое сообщение | до 64 KB |
| image | Изображение | до 10 MB |
| file | Произвольный файл | до 100 MB |
| system | Служебное (статус, событие группы) | до 1 KB |

Крупные файлы разбиваются на chunks и собираются на стороне получателя.

---

## Статусы сообщений

```
Отправитель видит:
  ○  отправлено    — сообщение ушло с устройства на узел
  ◑  доставлено    — узел получателя принял сообщение
  ●  прочитано     — получатель открыл сообщение

Технически:
  sent      → узел отправителя принял пакет (локальное подтверждение)
  delivered → узел получателя прислал ack
  read      → клиент получателя прислал read_receipt
```

Read receipts — опциональны. Ячейка может отключить отправку read_receipt в настройках.

## Групповые чаты

### Принцип
Группа — это отдельная сущность с собственным `group_id`.
Сообщение в группу отправляется один раз на узел, узел рассылает участникам.

### Структура группы
```
group_id   — уникальный идентификатор (UUID)
name       — название группы
owner      — cell_id создателя (единственный кто может удалить группу)
admins[]   — cell_id администраторов (могут добавлять/удалять участников)
members[]  — cell_id всех участников
home_node  — узел где "живёт" группа (хранит состав и историю доставки)
```

### Отправка сообщения в группу
```
1. Ячейка шифрует сообщение симметричным group_key
2. Отправляет на home_node группы
3. home_node рассылает всем участникам (на их home-узлы)
4. Если участник недоступен — буферизирует (TTL 72ч)
```

### Ротация group_key при выходе участника

При выходе или исключении участника group_key заменяется — чтобы вышедший не мог расшифровать новые сообщения.

```
1. Участник выходит (или owner/admin исключает)
2. home_node генерирует новый group_key
3. Шифрует новый group_key публичным ключом каждого оставшегося участника
4. Рассылает зашифрованные пакеты всем оставшимся
5. Все последующие сообщения шифруются новым ключом
```

Вышедший участник сохраняет доступ к переписке **до** момента выхода — Forward Secrecy в пределах членства.

### Управление группой
- Создать группу может любая ячейка
- Добавить участника — owner или admin
- Выйти из группы — любой участник сам
- Удалить группу — только owner
- При каждом выходе/исключении — автоматическая ротация group_key

## Аутентификация ячейки

### ZKP (протокол Шнорра)
```
1. Узел отправляет случайный challenge c
2. Ячейка генерирует r, отправляет R = g^r
3. Ячейка вычисляет s = r + c * x (x — приватный ключ)
4. Узел проверяет: g^s == R * X^c  (X — pubkey ячейки)
✓ Доказано: ячейка владеет приватным ключом, не раскрывая его
```

### MFA (TOTP)
- При первой регистрации: узел выдаёт TOTP-секрет
- Пользователь сохраняет в Google Authenticator или аналоге
- При каждом входе: ZKP + 6-значный TOTP код

---

## Ключи шифрования переписки

- **Signing keypair:** Ed25519 — подпись сообщений (идентификация)
- **Encryption keypair:** X25519 — шифрование содержимого (конфиденциальность)

При регистрации ячейка генерирует оба keypair.
Публичные ключи (Ed25519 + X25519) публикуются на home-узле.
Приватные ключи — только на устройстве пользователя.

---

## Смена home-узла (миграция)

Ячейка может сменить постоянный узел при условии согласования.

```
1. Ячейка договаривается с оператором нового узла (любым способом)
2. Новый узел выдаёт migration_token ячейке
3. Ячейка предъявляет migration_token старому home-узлу
4. Старый узел подписывает migration_cert:
     { cell_id, old_node, new_node, sig_old_node }
5. Новый узел регистрирует ячейку, принимает migration_cert
6. Gossip рассылает обновление: "cell_id теперь на node_NEW"
7. Все контакты автоматически получают новый адрес ячейки
8. История переписки на старом узле — не переносится (остаётся/удаляется)
```

Миграция требует участия обоих узлов — защита от принудительного переноса.

---

## Потеря ячейки

Если пользователь потерял ВСЕ устройства (master-ключ утерян):
- Восстановление невозможно — это сделано намеренно
- Нет "восстановления по email", нет "забыл пароль"
- Ячейка создаётся заново с новым keypair
- Контакты нужно уведомить о новом cell_id вручную
- Переписка утеряна (конфиденциальность важнее удобства)

---

## Требования к клиентскому приложению

- Хранит приватные ключи в защищённом хранилище ОС (Keychain, Keystore)
- Подключается к узлу по WebSocket TLS
- Работает на: Windows, macOS, Linux, Android, iOS
- Поддерживает переключение между узлами (home / visiting)
