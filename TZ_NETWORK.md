# ТЗ Сети (Network)

## Топология

Сеть Nevod — граф равноправных постоянных узлов без центра.
Каждый узел знает все остальные узлы через единую node_table.
Нет главных узлов — все постоянные узлы равны.

```
Node_1 ── Node_2 ── Node_5
   \          |        \
   Node_3 ── Node_4   Node_T (временный, привязан к Node_5)
```

---

## Генезис-узел (Node_0)

Первый узел сети создаётся вручную без консенсуса — консенсус не от кого получать.

```
1. Оператор (создатель сети) запускает: nevod init --genesis
2. Генерируется keypair
3. Создаётся node_table с единственной записью — самим собой
4. Узел начинает слушать входящие соединения
5. node_id первого узла публикуется создателем (сайт, лично, любым способом)
   — это единственная "точка доверия" в сети
```

Генезис-узел **не имеет привилегий** после создания сети.
Как только появляется второй узел — оба равноправны.
Если генезис-узел умрёт (72ч offline) — сеть продолжает работать без него.

---

## Node Table

Единая таблица узлов, которую имеет каждый узел сети.

```
Поля записи:
  node_id      — Ed25519 pubkey узла (уникальный идентификатор)
  address      — IP:port или домен
  type         — permanent | temporary
  access       — open | closed  (только для permanent)
  cert         — сертификат регистрации
  parent_node  — node_id пригласившего узла (только для temporary)
  last_seen    — timestamp последнего успешного пинга
  status       — online | offline | pending_removal
```

**access = open**: ячейки могут регистрироваться без invite-кода
**access = closed**: ячейки регистрируются только по invite-коду оператора

## Gossip Protocol — синхронизация сети

### Цикл (каждые 30 секунд на каждом узле)
```
1. Пинг всех узлов из node_table → обновить last_seen
2. Узел не ответил → increment missed_pings
3. missed_pings >= 3 → статус = offline
4. offline > 72 часов → статус = pending_removal
5. Обмен node_table с 3-5 случайными соседями (push-pull)
6. Распространить pending_removal по сети → все удаляют узел
```

### Распространение изменений
```
Событие (новый узел / узел умер) →
  узел рассылает update 3 случайным соседям →
    каждый из них рассылает ещё 3 соседям →
      за ~log(N) шагов весь граф получил обновление
```

## Регистрация нового постоянного узла

Процесс двухэтапный. Максимальное время: **24 часа**.

### Этап 1 — Рекомендация (социальный, ручной)

```
1. Оператор нового узла обращается к любому существующему
   участнику сети (лично, через сайт, любым способом)
2. Существующий узел-рекомендатель выдаёт invite_token:
     { recommended_node_id, recommender_node_id, sig, expires_in: 24h }
3. Без invite_token вход в сеть невозможен
```

Рекомендатель несёт репутационную ответственность за того кого пригласил.
Если рекомендованный узел ведёт себя плохо — это отражается на рекомендателе.

### Этап 2 — Консенсус (автоматический)

```
4. Новый узел генерирует keypair
5. Отправляет registration_request рекомендателю:
     { node_id, address, type: "permanent", invite_token }
6. Рекомендатель проверяет invite_token и рассылает заявку по сети (gossip)
7. Случайная выборка из 20 активных узлов получает запрос на голосование
8. Каждый из 20 проверяет:
     - invite_token валиден и не истёк
     - node_id уникален в node_table
     - address доступен (ping)
9. Узел голосует: подписывает одобрение или отказывает
10. Собрано 10+ подписей (50% из 20) → регистрация одобрена
11. node_cert рассылается по сети → все добавляют в node_table
12. Новый узел получает полную node_table от рекомендателя
```

### Защита от Sybil-атаки
- Без рекомендации войти нельзя — социальный барьер
- Рекомендатель может пригласить не более 3 узлов в 30 дней
- Один IP-адрес — один узел
- Консенсус из случайной выборки — нельзя предсказать кто будет голосовать

## Регистрация эфемерного узла

```
1. Home-узел генерирует invite_token (подписанный своим ключом)
2. Новый эфемерный узел предъявляет invite_token
3. Home-узел подписывает ephemeral_cert
4. Home-узел рассылает уведомление по сети:
     { type: "ephemeral_node", node_id, ephemeral_by: home_node_id }
5. Сеть добавляет в node_table (без голосования)
6. Эфемерный узел устанавливает постоянное WebSocket-соединение с home-узлом
   (якорная точка для hole punching и relay)
```

### Уничтожение эфемерного узла

**Явная команда (`nevod temp destroy`):**
```
1. Узел отправляет home-узлу: { type: "node_leaving", node_id }
2. Home-узел рассылает: { type: "node_remove", node_id } по сети
3. Все удаляют из node_table немедленно
4. Home-узел обновляет presence всех ячеек с этого узла
```

**Автоматически (TTL 72ч):**
```
1. Узел недоступен → missed_pings ≥ 3 → статус offline
2. offline > 72ч → pending_removal → сеть удаляет из node_table
   (данные на устройстве не удаляются автоматически)
```

### Связность за NAT — UDP hole punching

```
Временный узел за NAT (домашний роутер / мобильная сеть):

1. При старте: эфемерный узел сам подключается к home-узлу (исходящее соединение)
   Home-узел фиксирует: { node_id → external_ip:port } из NAT mapping

2. Node_X хочет достучаться до эфемерного:
   → запрашивает у home-узла: "дай адрес эфемерного"
   → home отвечает: { external_ip, external_port }
   → Node_X и эфемерный одновременно шлют UDP друг другу
   → NAT пробивается → прямое P2P соединение

3. Если hole punching не прошёл:
   → трафик идёт через home-узел как relay
   → home-узел видит только зашифрованные blob-ы
```

## Маршрутизация сообщений

### Алгоритм
```
Сообщение: from_cell@Node_A → to_cell@Node_X

1. Node_A проверяет presence_table: где сейчас to_cell?
   → Запрос на home-узел to_cell: "где сейчас cell_id?"
2. Home-узел отвечает: [Node_X (visiting), Node_H (home)]
3. Node_A пробует Node_X (visiting, приоритет)
4. Node_X недоступен → пробует Node_H (home)
5. Node_H недоступен → кладёт в message_buffer
```

### Прямое соединение
Если Node_A и Node_X знают друг друга → прямой WebSocket → минимальная задержка.

### Multi-hop (через промежуточные узлы)
```
Если Node_A не может напрямую достучаться до Node_X:
Node_A → Node_M (промежуточный) → Node_X
```
Выбор маршрута: минимальное количество хопов по node_table.

## Протокол сообщений

### Формат пакета (MessagePack)
```
{
  "v": 1,                          // версия протокола
  "id": "uuid-v4",                 // уникальный ID сообщения
  "type": "msg|ack|ping|gossip|...",
  "from": "cell_id@node_id",
  "to":   "cell_id@node_id",
  "payload": <bytes>,              // зашифрованный контент (E2E)
  "sig": <bytes>,                  // Ed25519 подпись отправителя
  "ts": 1714000000                 // unix timestamp
}
```

### Типы пакетов

| type | Описание |
|------|----------|
| `msg` | Сообщение от ячейки к ячейке |
| `ack` | Подтверждение доставки |
| `ping` | Проверка доступности узла |
| `pong` | Ответ на ping |
| `gossip` | Синхронизация node_table |
| `node_register` | Заявка на регистрацию узла |
| `node_vote` | Голос за регистрацию узла |
| `node_cert` | Сертификат зарегистрированного узла |
| `node_remove` | Уведомление об удалении узла |
| `presence_query` | Запрос местонахождения ячейки |
| `presence_response` | Ответ с местонахождением |
| `cell_register` | Регистрация новой ячейки на узле |
| `cell_invite` | Приглашение ячейки на visiting-узел |

## E2E шифрование

### Схема
```
Отправитель:
  1. Генерирует ephemeral X25519 keypair (одноразовый)
  2. Вычисляет shared_secret = X25519(ephemeral_secret, recipient_x25519_pubkey)
  3. Шифрует payload: ChaCha20-Poly1305(shared_secret, plaintext)
  4. Пакет: { ephemeral_pubkey, ciphertext, nonce }

Получатель:
  1. shared_secret = X25519(own_x25519_secret, ephemeral_pubkey)
  2. plaintext = ChaCha20-Poly1305.decrypt(shared_secret, ciphertext)
```

Узлы-посредники видят только зашифрованный blob и метаданные маршрутизации.

## Аутентификация узел↔узел (ZKP Шнорра)

```
При каждом установлении WebSocket-соединения между узлами:

Узел_A (Prover):             Узел_B (Verifier):
  генерирует r               
  R = g^r  ──────────────────→
                             генерирует challenge c
               ←────────────── c
  s = r + c * secret_a
  s  ──────────────────────→
                             проверяет: g^s == R * pubkey_A^c
                             ✓ Узел_A доказал владение ключом
```

Взаимная аутентификация: оба узла проходят процедуру в обе стороны.

## Параметры сети

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| Gossip интервал | 30 сек | Баланс актуальности и нагрузки |
| Ping timeout | 10 сек | Считать узел недоступным |
| Offline порог | 3 пропущенных пинга | ~90 сек до offline |
| TTL удаления узла | 72 часа | Время на перезагрузку/переезд |
| TTL буфера сообщений | 72 часа | Совпадает с TTL узла |
| Консенсус регистрации | 50% активных узлов | Byzantine fault tolerance |
| Порт по умолчанию | 8765 (WebSocket TLS) | |
| Макс. размер файла | 100 MB | Разбивается на chunks |
| Chunk size | 64 KB | |
