# Nevod — Концепция

## Идея

Nevod — децентрализованный защищённый мессенджер на основе сети равноправных узлов.
Название: "невод" — рыболовная сеть, где каждый узел держит другие.

Главный принцип: **конфиденциальность важнее сохранности данных.**
Вся информация сосредоточена в узлах — нет центрального сервера, нет единой точки уязвимости.

---

## Ключевые идеи

### Сеть равноправных узлов
Нет иерархии "главный — подчинённый". Все узлы равноправны, как в сети Биткоина.
Каждый новый узел регистрируется через голосование: нужно одобрение 50% активных узлов сети.
Каждый узел всегда имеет актуальную таблицу всех узлов сети.

### Вход в сеть — invite-based

Войти в сеть без рекомендации невозможно. Процесс двухэтапный (до 24 часов):

```
Этап 1 (ручной):   оператор нового узла получает invite_token
                   от любого существующего участника сети

Этап 2 (авто):     случайные 20 узлов голосуют → нужно 10+ подписей
                   → узел зарегистрирован → получает node_table
```

Рост сети:
```
Узел_1 (корневой, создаётся вручную)
Узел_2 → рекомендован Узлом_1 → консенсус (20 выборка)
Узел_3 → рекомендован любым → консенсус
...
```

Один рекомендатель может пригласить не более 3 новых узлов за 30 дней.

### Ячейки — пользователи сети
Пользователь = ячейка. Ячейка живёт на узле, но не привязана к нему навсегда.
Идентичность ячейки — это её криптографический ключ (Ed25519), а не адрес на узле.

Адрес ячейки: `cell_id@node_id` — указывает текущий контекст, не постоянный адрес.

### Два типа узлов

| Тип | Регистрация | Жизненный цикл |
|-----|-------------|----------------|
| Постоянный | 50% консенсус сети | Долгосрочный |
| Эфемерный | Приглашение от home-узла | Часы / дни, явное уничтожение |

**Пример эфемерного узла:**
Семья едет в Таиланд. На телефоне поднимается Node_A для локального общения
и связи с родными через Node_B (постоянный, Киев). Поездка закончилась —
Node_A уничтожается. Все локальные переписки исчезают навсегда. Это не баг, это фича.

### Маршрутизация сообщений

**Внутри одного узла:**
```
Ячейка_А ↔ Ячейка_Б  (прямая доставка, без интернета если локальная сеть)
```

**Между узлами:**
```
Ячейка → Узел_А → ... → Узел_Х → Ячейка
```

Каждый узел знает все остальные узлы (node_table) и строит маршрут самостоятельно.

### Присутствие (Presence)

Home-узел ячейки всегда знает где она сейчас находится:
```
alice → [Node_B (home★), Node_A (visiting)]
```

Поиск ячейки:
1. Запрос на home-узел → получаем список узлов где она есть
2. Пробуем эфемерный (если есть) → если жив, доставляем туда
3. Если эфемерный недоступен → доставляем на home
4. Если home недоступен → буферизируем у себя

### Буферизация сообщений

Любой узел буферизирует сообщения для недоступных адресатов.
Узел хранит зашифрованный blob — содержимого не видит.
TTL буфера: 72 часа (совпадает со временем удаления мёртвого узла).
Если узел-буфер уничтожен — сообщения исчезают. Конфиденциальность сохранена.

### Удаление узла из сети

Узел недоступен 72 часа → автоматически удаляется из node_table всех узлов.
Gossip-протокол распространяет обновление по всей сети.

### Безопасность

- **Идентификация узлов и ячеек:** Ed25519 keypair
- **Аутентификация:** ZKP (протокол Шнорра) — доказывает владение ключом без его раскрытия
- **MFA для ячеек:** ZKP + TOTP (одноразовый код)
- **E2E шифрование:** ячейка шифрует сообщение публичным ключом получателя (X25519 + ChaCha20)
- **Узлы не видят содержимое** — только зашифрованные blob-ы и метаданные маршрутизации
- **Транспорт:** WebSocket over TLS

---

## Чего Nevod не делает

- Не хранит данные централизованно
- Не требует телефонного номера или email для регистрации
- Не позволяет восстановить переписку после уничтожения узла
- Не раскрывает содержимое сообщений узлам-посредникам

## Технологический стек (план)

| Компонент | Технология |
|-----------|-----------|
| Язык сервера | Python (asyncio) |
| Транспорт | WebSocket over TLS (websockets lib) |
| Криптография | PyNaCl (libsodium): Ed25519, X25519, ChaCha20 |
| ZKP | Schnorr protocol (реализация своя) |
| MFA | pyotp (TOTP) |
| Хранилище узла | SQLite |
| Протокол сообщений | MessagePack (бинарный, компактный) |
| Node table sync | Gossip protocol (push-pull каждые 30 сек) |

## Вдохновение и источники

- Принцип сети: Bitcoin P2P

- Транспорт: WebSocket (RFC 6455)
