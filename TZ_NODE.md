# ТЗ Узла (Node)

## Определение

Узел — базовая единица инфраструктуры сети Nevod.
Программа, которая запускается на устройстве (сервер, ПК, телефон) и обеспечивает:
- хранение ячеек и их данных
- маршрутизацию сообщений
- буферизацию для оффлайн-адресатов
- участие в сети (gossip, консенсус)

---

## Идентификация узла

- **node_id** — Ed25519 публичный ключ узла (генерируется при создании)
- **node_secret** — Ed25519 приватный ключ (хранится только локально, никогда не передаётся)
- **node_cert** — сертификат подписанный 50% узлов сети при регистрации
- **node_address** — IP:port (или домен) для входящих соединений

---

## Типы узлов

### Постоянный узел
- Регистрируется через двухэтапный процесс (рекомендация + консенсус), до 24ч
- Долгосрочный, стабильный адрес
- Хранит home-ячейки и presence table
- Равноправен с другими постоянными узлами — нет главных
- Делится на закрытый и открытый (см. ниже)

### Временный узел (бывший "эфемерный")
- Создаётся по приглашению существующего постоянного узла
- Постоянный узел подписывает сертификат временного
- Остальная сеть получает уведомление (без голосования)
- **Уничтожение:** явная команда `nevod temp destroy` ИЛИ автоматически по TTL (72ч недоступности)
- НЕ может регистрировать постоянные ячейки — только принимать visiting-ячейки
- НЕ может выдавать рекомендации для новых постоянных узлов

#### Связность временного узла за NAT (UDP hole punching)
Временный узел может находиться за NAT (домашний роутер, мобильная сеть).
Входящие соединения напрямую невозможны — используется UDP hole punching:

```
1. Временный узел при старте подключается к home-узлу (исходящее соединение)
2. Home-узел знает внешний IP:port временного узла (из NAT mapping)
3. Когда Node_X хочет достучаться до временного узла:
   Node_X → запрашивает у home-узла внешний адрес временного
   Home-узел → отвечает: { external_ip, external_port }
   Node_X + временный узел → одновременно шлют UDP-пакеты друг другу
   NAT временного узла "пробивается" → прямое P2P соединение
4. Если hole punching не прошёл → трафик идёт через home-узел как relay
```

Временный узел **обязан** поддерживать постоянное соединение с home-узлом — это якорная точка для hole punching.

## Закрытый и открытый постоянный узел

Узел — это одновременно **инфраструктура** и **сообщество**.

### Закрытый узел
- Вступить можно только по invite-коду от оператора
- Список участников виден только зарегистрированным участникам
- Оператор полностью контролирует состав
- Подходит для: семьи, организации, доверенного круга
- Аналог: закрытый чат

### Открытый узел
- Вступить можно без приглашения
- Список участников виден всем в сети (включая не-участников)
- Узел имеет название, описание и тему
- Подходит для: сообществ по интересам, точек знакомств
- Аналог: публичный Telegram-канал/группа по интересам
- Оператор может закрыть узел в любой момент (новые регистрации прекращаются,
  существующие ячейки остаются)

### Как Алиса видит мир

```
Алиса зарегистрирована на Node_A (закрытый, семья):
  → видит список участников Node_A

Алиса просматривает открытые узлы сети:
  → видит каталог открытых узлов (название, тема, кол-во участников)
  → заходит на Node_B (открытый, "Фотографы Киева")
  → видит список участников Node_B
  → находит Боба → добавляет в контакты → пишет напрямую

Переписка Алисы с Бобом идёт напрямую через сеть,
не через Node_B — узел только точка знакомства.
```

### Каталог открытых узлов
- Каждый открытый узел публикует: `{ node_id, name, topic, member_count }`
- Эти данные распространяются через gossip и видны всей сети
- Алиса видит каталог в клиентском приложении и выбирает сообщества

Тип узла (открытый/закрытый) указывается в node_table и виден всей сети.

---

## Внутренние структуры данных

### node_table — таблица всех узлов сети
```
node_id | address | type | cert | last_seen | status
```
- Обновляется через gossip каждые 30 секунд
- Узел помечается offline если нет ответа 3 пинга подряд
- Узел удаляется если offline > 72 часов

### cells — ячейки этого узла
```
cell_id | pubkey | registered_at | home (bool)
```

### presence_table — где сейчас находятся home-ячейки
```
cell_id | home_node_id | visiting_node_id | updated_at
```

### message_buffer — буфер сообщений для недоступных адресатов
```
message_id | to_cell | to_node | payload (encrypted) | created_at | ttl
```
- TTL: 72 часа
- При истечении TTL сообщение удаляется без доставки
- Узел хранит только зашифрованный blob — содержимое недоступно

## Функции узла

### 1. Регистрация в сети
- Генерация keypair
- Отправка заявки известным узлам
- Сбор подписей (50% консенсус)
- Получение node_table
- Рассылка своего присутствия (gossip)

### 2. Gossip — синхронизация node_table
- Каждые 30 секунд: обменивается node_table с соседними узлами
- Push: отправляет изменения которых нет у соседа
- Pull: запрашивает изменения которых нет у себя
- Пингует все известные узлы, обновляет last_seen

### 3. Маршрутизация сообщений
- Получает сообщение → смотрит to_node в node_table
- Прямое соединение с to_node → отправляет напрямую
- Нет прямого → выбирает промежуточный узел (минимум хопов)
- to_node недоступен → кладёт в message_buffer

### 4. Буферизация
- Хранит зашифрованные сообщения для недоступных адресатов
- Проверяет каждые N минут — не появился ли адресат
- При появлении адресата — доставляет и удаляет из буфера
- TTL истёк — удаляет без доставки

### 5. Presence — реестр местонахождения ячеек
- Когда ячейка подключается к эфемерному узлу — home-узел обновляет presence_table
- Отвечает на запросы других узлов: "где сейчас cell_id?"

### 6. Аутентификация
- **Узел↔Узел:** ZKP (протокол Шнорра) — взаимная идентификация без передачи ключей
- **Ячейка→Узел:** ZKP + TOTP (MFA)

## Безопасность узла

- Приватный ключ генерируется локально, никогда не покидает устройство
- TLS на всех WebSocket соединениях
- ZKP при каждом установлении соединения узел↔узел
- Содержимое сообщений недоступно — только зашифрованные blob-ы
- При уничтожении узла: удалить SQLite базу, удалить keypair

---

## Администрирование узла

Принцип: **максимальная защищённость** — минимальная поверхность атаки.

### Постоянный узел — CLI + конфиг-файл
- Управление через командную строку (нет веб-интерфейса = нет веб-атак)
- Конфиг-файл `nevod.conf` для базовых настроек
- Команды: `nevod status`, `nevod cell list`, `nevod cell ban`, `nevod node table`
- Логи в файл (не в сеть)

### Временный узел — упрощённое управление
- Тот же CLI, но с ограниченным набором команд
- Нет команд управления ячейками — временный узел не регистрирует постоянных ячеек
- Создаётся и уничтожается одной командой: `nevod temp create` / `nevod temp destroy`

#### Процедура `nevod temp destroy`
```
1. Узел уведомляет home-узел: "я уничтожаюсь"
2. Home-узел рассылает по сети: { type: "node_remove", node_id }
3. Все узлы удаляют из node_table
4. Home-узел обновляет presence_table всех ячеек которые были на этом узле
5. Зашифрованный SQLite удаляется с диска
6. Keypair удаляется
```

Автоматическое уничтожение: если узел недоступен 72ч — gossip помечает `pending_removal` и сеть удаляет его без явной команды. Данные на устройстве остаются (оператор должен удалить вручную).

## Требования к окружению

| | Постоянный узел | Временный узел |
|-|----------------|----------------|
| Python | 3.11+ | 3.11+ |
| RAM | минимум 512MB | минимум 256MB |
| Диск | минимум 10GB | минимум 1GB |
| Сеть | статический IP или домен | любой IP |
| Порт | 8765 (открытый наружу) | 8765 (может быть за NAT) |
| Платформы | Linux, Windows, macOS | + Android (Termux) |
