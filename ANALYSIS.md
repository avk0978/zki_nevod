# Анализ существующих проектов (обновлённый)

> Обновлено с учётом финальной концепции Nevod:
> временные узлы + UDP hole punching, открытые/закрытые узлы,
> multi-device, ротация group_key, явное уничтожение узла,
> архитектурная гарантия нулевого доступа разработчика.

## Проекты для анализа

| Проект | Тип | Год | Статус |
|--------|-----|-----|--------|
| Matrix/Element | Федерация | 2014 | Активен, крупнейший |
| Session | Децентрализованный | 2018 | Активен, ~1M users |
| SimpleX | Децентрализованный | 2022 | Активен, быстро растёт |
| Briar | P2P | 2013 | Активен, нишевый |
| Jami | P2P DHT | 2016 | Активен |
| Tox | P2P DHT | 2013 | Медленно развивается |
| Nostr | Relay-based | 2020 | Активен, соцсеть > мессенджер |

## 1. Matrix / Element

**Архитектура:**
- Федерация homeserver-ов
- Пользователь: `@alice:matrix.org` — ID привязан к серверу
- E2E для личных чатов (Olm, Double Ratchet) и групп (Megolm, ratchet)
- Сервер хранит состав комнат, метаданные, и незашифрованные события (если E2E не включён)

**Плюсы:**
- Зрелый протокол, E2E (Megolm), голос/видео, мосты к другим сетям
- Федерация: много операторов, нет единой точки отказа
- Cross-signing для multi-device
- Spaces — иерархия сообществ (аналог Slack)

**Минусы:**
- Данные на homeserver видны его администратору
- Тяжёлый (Synapse: 1+ GB RAM)
- Homeserver видит состав комнат, метаданные участников
- ID пользователя навсегда привязан к серверу

**Сравнение с Nevod:**

| Аспект | Matrix | Nevod |
|--------|--------|-------|
| Синхронизация | Federation protocol | Gossip (push-pull, 30 сек) |
| Данные при падении сервера | Сохраняются → риск утечки | Исчезают навсегда |
| ID пользователя | @user:server (привязан) | cell_id (keypair, переносимый) |
| Группы E2E | Megolm (ratchet, сложный) | group_key + ротация при выходе |
| Multi-device | Cross-signing (зрелое) | master keypair + device certs |
| Сообщества | Spaces + комнаты | Открытые узлы (каталог) |
| Временные пространства | Нет | Временные узлы |
| Доступ оператора | Видит метаданные, состав | Только зашифрованный blob |
| Администрирование | Web UI | CLI only |

**Ключевое отличие:** Matrix сохраняет всё на сервере и этот сервер можно взломать, получить ордер или отключить. В Nevod данные исчезают с узлом — нет смысла взламывать то, чего нет.

**Что берём:** паттерны federation routing, концепция rooms → открытые узлы

## 2. Session

**Архитектура:**
- Decentralised Service Nodes
- Стейкинг 15,000 OXEN (ранее 25,000 SESH) — финансовый залог оператора
- Onion routing: 3 слоя анонимизации (как Tor)
- Swarm-группы: сообщение хранится на 5 случайных узлах swarm до 14 дней
- Пользователь = публичный ключ (Session ID), без телефона и email

**Плюсы:**
- Нет телефона, нет email — ближайший аналог Nevod в этом плане
- Onion routing — метаданные отправителя скрыты даже от узлов
- Реальная децентрализация через финансовый барьер

**Минусы:**
- Нужна криптовалюта для запуска узла — высокий барьер ($5–15K)
- Сообщения хранятся 14 дней в swarm — противоречит принципу «данные умирают»
- Зависимость от блокчейна Oxen
- Нет открытых сообществ / каталога для знакомств

**Сравнение с Nevod:**

| Аспект | Session | Nevod |
|--------|---------|-------|
| Входной барьер для узла | $5–15K стейкинг | Социальная рекомендация |
| Анонимность маршрута | Onion routing (3 слоя) | Прямой P2P (метаданные видны узлам) |
| Хранение сообщений | 14 дней в swarm | 72 ч в буфере, умирает с узлом |
| Без телефона/email | Да | Да |
| Сообщества / discovery | Нет | Открытые узлы + каталог |
| Временные узлы | Нет | Да (poездка, мероприятие) |
| Зависимость от блокчейна | Да | Нет |

**Ключевое отличие:** Session защищает *кто с кем говорит* (onion routing). Nevod защищает *что было сказано* (данные умирают с узлом). Разные модели угрозы. Nevod проще войти, но слабее против анализа трафика.

**Что берём:** идея swarm-resilience, защита от Sybil без финансового стейкинга

## 3. SimpleX Chat

**Архитектура:**
- **Нет глобального user ID вообще** — ни телефона, ни username, ни keypair
- Связь через simplex queues — односторонние очереди, у каждого контакта разная пара
- Relay-серверы не видят граф связей: Алиса у Боба — один ID, у Карла — другой
- Сервер не хранит сообщения после доставки

**Плюсы:**
- Максимальная защита метаданных: сервер не может построить граф «кто с кем»
- Разные ID для каждого контакта — потеря одного не раскрывает других
- Можно поднять свой relay

**Минусы:**
- Нет постоянного адреса — нельзя написать незнакомцу без обмена ссылкой
- Нет community discovery / каталога
- Нет presence — offline-буфера по умолчанию нет
- Нет временных узлов

**Сравнение с Nevod:**

| Аспект | SimpleX | Nevod |
|--------|---------|-------|
| Глобальный ID | Нет вообще | cell_id (keypair) |
| Метаданные на сервере | Сервер не видит граф | Узел видит маршрутизацию |
| Discovery незнакомцев | Невозможен | Открытые узлы + каталог |
| Offline-буфер | Нет | 72 ч |
| Presence | Нет | Да |
| Временные узлы | Нет | Да |
| Сложность UX | Высокая | Средняя |

**Ключевое отличие:** SimpleX выигрывает по защите метаданных связей. Nevod выигрывает по usability и discovery. Если для SimpleX нужно заранее обменяться ссылкой, то в Nevod незнакомцы могут найти друг друга через открытые узлы.

**Что берём:** идея «нет глобального ID» как цель, очереди как транспортный паттерн

## 4. Briar

**Архитектура:**
- Истинный P2P — нет серверов вообще
- Транспорты: Tor (интернет), WiFi (локально), Bluetooth (оффлайн)
- Gossip protocol для синхронизации сообщений — тот же подход что в Nevod
- Контакты добавляются через QR-код (при встрече)

**Плюсы:**
- Работает без интернета (WiFi / Bluetooth)
- Gossip — хорошо изученная технология (используется в Nevod)
- Нет серверов = нечего взламывать

**Минусы:**
- Оба пользователя онлайн одновременно (нет offline-буфера без Tor)
- Медленная синхронизация через Tor
- Нет поддержки нескольких устройств
- Нет community discovery

**Сравнение с Nevod:**

| Аспект | Briar | Nevod |
|--------|-------|-------|
| Инфраструктура | Нет совсем | Узлы (необходимы) |
| Offline без интернета | BT + WiFi | Нет (только буфер 72ч) |
| Gossip | Да | Да (30 сек цикл) |
| Offline-буфер (с инт.) | Нет | 72 ч |
| Multi-device | Нет | Да (master + device keypair) |
| Discovery | QR при встрече | Открытые узлы + каталог |
| Временные узлы | Нет | Да |

**Ключевое отличие:** Briar — максимальный оффлайн без инфраструктуры. Nevod — асинхронность и community через узлы. Briar лучше для протестов и зон без связи. Nevod лучше для повседневного доверенного общения.

**Что берём:** gossip protocol (уже в ТЗ Nevod), философия offline-first

## 5. Jami

**Архитектура:**
- OpenDHT для обнаружения пользователей (Kademlia)
- Аккаунт = Ed25519 ключ (как в Nevod)
- P2P соединение напрямую между устройствами
- TURN-серверы как fallback (аналог relay в Nevod)
- Синхронизация нескольких устройств через DHT

**Плюсы:**
- Ed25519 идентичность (совпадает с Nevod)
- Видеозвонки из коробки
- Работает на мобильных

**Минусы:**
- DHT требует что хотя бы один bootstrap-узел всегда доступен
- Сложность правильной реализации DHT
- Нет community discovery

**Сравнение с Nevod:**

| Аспект | Jami | Nevod |
|--------|------|-------|
| Discovery узлов | Kademlia DHT | Full node_table (gossip) |
| Идентичность | Ed25519 | Ed25519 |
| NAT traversal | TURN / ICE | UDP hole punching + relay |
| Видеозвонки | Да | Нет (пока) |
| Сообщества | Нет | Открытые узлы |
| Временные узлы | Нет | Да |

**Что берём:** DHT как вдохновение, Ed25519 identity — совпадает

## 6. Tox

**Архитектура:**
- P2P DHT, нет серверов
- Tox ID = публичный ключ
- Нет offline-доставки в оригинальном протоколе

**Плюсы:** простота, нет серверов
**Минусы:** получатель должен быть онлайн, медленное развитие

**Что берём:** простота keypair-based identity

## 7. Nostr

**Архитектура:**
- Пользователь = keypair, публикует события на relay-серверы
- Relay = простой WebSocket сервер
- Нет E2E по умолчанию (DM через NIP-04 — слабоватое шифрование)
- Больше соцсеть, чем мессенджер

**Плюсы:** простота протокола, WebSocket-транспорт (совпадает с Nevod), event-driven

**Что берём:** WebSocket-протокол, event-driven модель

## Честный разбор: где Nevod слабее конкурентов

### 1. Анонимность маршрута — слабее Session
Session использует onion routing (3 слоя): ни один узел не знает одновременно отправителя и получателя.
В Nevod узлы-посредники видят метаданные маршрутизации (`from_node`, `to_node`).
**Вывод:** Nevod защищает содержимое, но не скрывает факт общения между узлами.

### 2. Forward Secrecy — слабее Signal
Signal использует Double Ratchet: каждое сообщение шифруется новым ключом, и компрометация одного не раскрывает предыдущие и последующие.
Nevod использует ephemeral X25519 per message — это обеспечивает forward secrecy, но без "break-in recovery": если долгосрочный X25519 ключ ячейки скомпрометирован, злоумышленник может вычислить будущие ephemeral secrets.
**Вывод:** Для большинства сценариев достаточно. Для высокорисковых (журналисты) Signal лучше.

### 3. Метаданные связей — слабее SimpleX
SimpleX не имеет глобального ID: сервер не может построить граф «кто с кем». В Nevod cell_id глобален — узлы знают кто с кем общается (но не содержимое).
**Вывод:** Если цель — скрыть сам факт знакомства, SimpleX лучше. Nevod не ставит такой цели.

### 4. Экосистема — слабее всех
Ни ботов, ни каналов, ни интеграций, ни истории разработки.
**Вывод:** Ожидаемо для нового проекта. Не является проектной целью.

### 5. Работа без интернета — слабее Briar
Briar работает по Bluetooth и WiFi. Nevod требует интернет.
**Вывод:** Разные сценарии использования. Для повседневного использования не критично.

## Уникальность Nevod (подтверждена сравнением)

После анализа всех 7 проектов ни один не реализует следующую комбинацию:

### 1. Архитектурная гарантия нулевого доступа разработчика
Ни Telegram, ни Matrix не могут дать эту гарантию — у них есть сервера.
Signal технически близок, но централизованный сервер Signal Foundation существует.
В Nevod разработчик не владеет ни одним узлом сети. **Требовать ключи не имеет смысла — их физически не существует.**

### 2. Временные узлы + явное уничтожение
`nevod temp destroy` — команда уничтожения с удалением SQLite + keypair.
**Ни один проект** не имеет концепции «временного узла для поездки» с гарантированным удалением данных по команде.

### 3. Данные умирают с узлом
Matrix хранит бессрочно. Session — 14 дней. SimpleX — до доставки. Nevod — 72ч + умирают при уничтожении узла. Это наиболее радикальная позиция по конфиденциальности.

### 4. Открытые узлы как community hubs
Каталог сообществ через gossip, видимый всей сети. Переписка идёт напрямую — узел только точка знакомства. Telegram-каналы привязаны к Telegram. Невод-сообщества — к распределённой сети без центра.

### 5. Демократический консенсус без финансового барьера
Session: $5–15K стейкинг. Nostr: любой. Nevod: социальная рекомендация + 50% голосование. Защита от Sybil без криптовалюты.

### 6. ZKP (Шнорр) для аутентификации узел↔узел
Протокол Шнорра между узлами не применяется нигде из рассмотренных проектов.

### 7. Presence-система
Home-узел знает где сейчас находится ячейка (home / visiting). Ни один из рассмотренных проектов не имеет аналога.

## Целевая аудитория vs конкуренты

| Сценарий | Лучший выбор сейчас | Почему Nevod лучше |
|----------|--------------------|--------------------|
| Семья / близкий круг | Telegram (закрытый чат) | Закрытый узел: данные не у Telegram |
| Организация / корпоратив | Slack / Matrix | Узел на своём сервере, нет зависимости |
| Журналист / активист | Signal | ZKP + нет телефона; но Signal = централизованный сервер |
| Поездка / мероприятие | Telegram-группа | Временный узел: по возвращении всё исчезает |
| Анонимное общение | Session / SimpleX | Есть открытые узлы, но нет onion routing |
| Сообщество по интересам | Telegram-канал | Открытый узел без зависимости от компании |

## Риски выявленные в ходе анализа

### 1. Bootstrap проблема — РЕШЕНА ✓
Invite-based модель. Новый узел входит только по рекомендации.
Один рекомендатель = не более 3 приглашений за 30 дней.

### 2. Масштабирование консенсуса — РЕШЕНА ✓
Голосует не вся сеть, а случайная выборка 20 узлов. 10 из 20 = 50%.

### 3. Sybil-атака — РЕШЕНА ✓
Социальный барьер + лимит приглашений + один IP = один узел.

### 4. Временный узел за NAT — РЕШЕНА ✓
UDP hole punching через home-узел как координатор. Fallback: relay через home.

### 5. Ротация ключей в группах — РЕШЕНА ✓
При выходе участника home_node генерирует новый group_key и рассылает оставшимся.

### 6. Уничтожение данных — РЕШЕНА ✓
`nevod temp destroy`: удаление SQLite + keypair + уведомление сети.
TTL 72ч как автоматический fallback.

### 7. Анонимность маршрута — ОТКРЫТА
Невод не скрывает метаданные маршрутизации между узлами.
**Решение (будущее):** onion-обёртка поверх существующей маршрутизации.

### 8. Node table при масштабировании — ОТЛОЖЕНА
10 000+ узлов = тяжёлый gossip.
**Решение (будущее):** переход на DHT при достижении предела.
