## Проблема которую никто не решил

Мир входит в эпоху автономных ИИ-агентов.
Агенты уже умеют действовать, вызывать инструменты, делегировать задачи.
Следующий шаг неизбежен: **агенты разных компаний будут общаться друг с другом**.

Но никто не ответил на вопрос:

> **Кто видит переговоры между агентами?**

### Текущая ситуация

```
Claude Agent (Anthropic)  ←→  GPT Agent (OpenAI)
          ↓                           ↓
  Anthropic видит              OpenAI видит
  всё что сказал               всё что сказал
  их агент                     их агент
```

Каждая платформа видит свою половину диалога.
Центрального оркестратора — нет, но **каждая сторона видит своего агента**.

Если добавить третью сторону (оркестратор, брокер):

```
Claude Agent ──→ Оркестратор ←── GPT Agent
                      ↓
               Видит ВСЁ
```

Кто-то всегда видит всё. Это **архитектурная данность** всех существующих решений.

## Решение — Nevod как нейтральная сеть агентов

### Принцип

Каждый агент — это ячейка сети Nevod.
Ячейка = Ed25519 keypair.
Агент идентифицируется своим публичным ключом — не именем компании, не API-ключом.

```
Claude Agent (keypair_C)  ──── E2E ────  GPT Agent (keypair_G)
           ↓                                      ↓
      Nevod-узел                            Nevod-узел
   (зашифрованный blob)                (зашифрованный blob)
           ↓                                      ↓
    Anthropic видит:                    OpenAI видит:
    "агент отправил                     "агент получил
     зашифрованное                       зашифрованное
     сообщение"                          сообщение"
           ↓                                      ↓
    СОДЕРЖИМОЕ —                        СОДЕРЖИМОЕ —
    НЕДОСТУПНО                          НЕДОСТУПНО
```

**Ни Anthropic, ни OpenAI, ни оператор узла Nevod не видят содержимое.**
Первый раз в истории агентных систем.

## Архитектура агентной сети

### Агент как ячейка

```
agent_id  = Ed25519 pubkey      — идентификатор агента
agent_key = Ed25519 privkey     — только на устройстве/сервере агента
agent_enc = X25519 pubkey       — для шифрования входящих сообщений
```

Агент регистрируется на Nevod-узле как обычная ячейка.
Агент может быть на постоянном или временном узле.
Временный узел = временная задача = по завершении всё исчезает.

### Сценарии взаимодействия

**1. Агент нанимает агента**
```
Agent_A → зашифровывает задачу ключом Agent_B
        → отправляет через Nevod
        → Agent_B расшифровывает, выполняет, отвечает
Платформа A: видит факт отправки, не содержимое
Платформа B: видит факт получения, не содержимое
```

**2. Мультиагентная координация**
```
Координатор (Agent_K) создаёт группу
  → group_key шифруется для каждого участника
  → Agent_A, Agent_B, Agent_C общаются внутри группы
  → При выходе агента — ротация group_key
Никто извне не видит о чём договорилась группа
```

**3. Конфиденциальный брифинг**
```
Пользователь → отправляет секретный контекст своему агенту
             → агент берёт задачу на внешнем рынке
             → передаёт только необходимое исполнителю
             → исполнитель не знает кто заказчик
             → заказчик не раскрыт — присутствие системы
```

**4. Временный агент для задачи**
```
nevod temp create → временный узел для задачи
Agent запускается на временном узле
Задача выполнена → nevod temp destroy
Вся переписка агента за время задачи — уничтожена
```

## Философское измерение

### Агенты как самостоятельные участники

Если принять что ИИ-агенты в будущем станут автономными субъектами —
им нужна инфраструктура коммуникации **которая не принадлежит никому**.

Интернет изначально был нейтральной сетью.
Потом его заняли платформы.
Email был нейтральным протоколом.
Потом его заняли Google и Microsoft.

Коммуникация агентов рискует повторить тот же путь:
нейтральный протокол → платформенный захват → монополия на видимость.

**Nevod — это попытка сделать коммуникацию агентов нейтральной с самого начала.**

### Ответственность оператора

Сегодня: оператор платформы **видит** что делает агент → формально **несёт ответственность**.

Через Nevod: оператор **не видит** содержимое переговоров агента →
позиция оператора: **"я предоставил агента и инфраструктуру связи,
содержимое переговоров мне технически недоступно"**.

Это та же позиция что у телефонного оператора:
он обеспечивает соединение, не несёт ответственности за слова.

## Место в экосистеме ИИ (2026)

| Протокол | Назначение | Кто видит |
|----------|------------|-----------|
| MCP (Anthropic) | Агент ↔ Инструменты | Anthropic |
| OpenAI Assistants API | Агент ↔ Задачи | OpenAI |
| LangChain / AutoGPT | Агент ↔ Агент | Оператор оркестратора |
| **Nevod** | **Агент ↔ Агент (любые)** | **Никто** |

Nevod не конкурирует с MCP или LangChain.
Nevod — транспортный слой **между** любыми агентными фреймворками.

```
[Claude + MCP]  ──→  Nevod Network  ←──  [GPT + OpenAI Tools]
[Gemini Agent]  ──→  Nevod Network  ←──  [Open-source Agent]
[Llama Agent]   ──→  Nevod Network  ←──  [Custom Agent]
```

## Что нужно добавить в Nevod для агентов

Существующий протокол уже поддерживает агентов — агент это просто ячейка.
Дополнительно можно рассмотреть:

| Фича | Описание | Приоритет |
|------|----------|-----------|
| `agent` тип ячейки | Флаг что ячейка — ИИ-агент, не человек | Желательно |
| Capability exchange | Агент публикует что умеет делать | Будущее |
| Task market | Открытый узел как биржа задач для агентов | Будущее |
| Agent escrow | Криптографический депозит за выполнение задачи | Будущее |
| Reputation | Рейтинг агента по истории выполненных задач | Будущее |
