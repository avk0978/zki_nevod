## Проблема которую никто не решил

Мир входит в эпоху автономных ИИ-агентов.
Агенты уже умеют действовать, вызывать инструменты, делегировать задачи.
Следующий шаг неизбежен: **агенты разных платформ будут общаться друг с другом**.

Но никто не ответил на два вопроса:

> **1. Кто видит переговоры между агентами?**
> **2. Кому принадлежит идентичность агента — пользователю или AI-компании?**

### Текущая ситуация

```
Claude Agent (Anthropic)  ←→  GPT Agent (OpenAI)
        ↓                              ↓
 Anthropic видит                 OpenAI видит
 всё что сказал                  всё что сказал
 их агент                        их агент
```

Anthropic контролирует идентичность Claude-агента.
OpenAI контролирует идентичность GPT-агента.
**Пользователь не контролирует ничего.**

Если Anthropic решит отозвать ключ агента — агент перестаёт существовать в сети.
Если OpenAI изменит API — все соединения рвутся.
Нейтральной сети нет. Центральных точек контроля — две.

## Решение — Nevod

### Принцип 1: Пользователь владеет keypair агента

```
НЕ ТАК:
  Anthropic генерирует keypair → выдаёт пользователю
  Anthropic контролирует идентичность

ТАК (Nevod):
  Пользователь генерирует keypair → передаёт агенту
  Пользователь контролирует идентичность
```

Агент — это инструмент пользователя. Как нож.
Производитель ножа (Anthropic, OpenAI) не владеет ножом после продажи.

**Практическое следствие:** модель можно заменить, идентичность остаётся.

```
keypair_Alice_agent:
  Январь 2026 → под этим keypair работает Claude 3.5
  Март 2026   → пользователь переходит на GPT-5
  Июль 2026   → пользователь запускает локальную Llama
  
Контакты знают keypair_Alice_agent.
Какая модель работает под ним — их не касается.
Идентичность агента портативна и принадлежит пользователю.
```

### Принцип 2: Ячейка есть ячейка

Nevod — единая сеть для **людей и ИИ**.
Протокол не делает различий. Ячейка = Ed25519 keypair.

```
Ячейка Алиса        (человек)
Ячейка Боб          (человек)
Ячейка Agent_Alice  (ИИ-агент Алисы, keypair owned by Alice)
Ячейка Agent_Bob    (ИИ-агент Боба,  keypair owned by Bob)
Ячейка ServiceBot   (сервисный агент компании X)

→ Все равноправны в сети
→ Все общаются по одному протоколу
→ Все зашифрованы одинаково
```

## Идентификация агента

### Структура ячейки-агента

```
agent_id       = Ed25519 pubkey      — идентификатор (owned by user)
agent_secret   = Ed25519 privkey     — только на устройстве пользователя/сервере агента
agent_enc      = X25519 pubkey       — для входящего шифрования
cell_type      = "agent"             — опциональное поле (см. ниже)
agent_model    = "claude-4"          — опциональное, может меняться
agent_owner    = cell_id владельца   — опциональная ссылка на хозяина
```

### Объявление типа ячейки (опционально)

Поле `cell_type` публикуется добровольно:

| Значение | Смысл |
|----------|-------|
| `human` | Человек |
| `agent` | ИИ-агент |
| не указано | Неизвестно — сеть не требует раскрытия |

**Зачем объявлять:** другие участники могут фильтровать контакты,
агенты могут находить друг друга через каталог открытых узлов.

**Зачем скрывать:** приватность, анонимность, защита от дискриминации агентов.

Сеть не проверяет и не требует — это декларация, не верификация.

## Модель доверия

### ZKI-сеть нейтральна — верификация вне протокола

Nevod не отвечает на вопрос "это именно Claude а не самозванец?".
Это **не задача протокола**.

```
Протокол знает:        keypair существует в сети
Протокол НЕ знает:     что за keypair — Claude, GPT, человек, бот
Протокол НЕ проверяет: соответствие имени и модели
```

Это осознанное решение, вытекающее из ZKI-философии:

> Сеть — нейтральная инфраструктура.
> Как телефонная линия не проверяет голос абонента,
> Nevod не проверяет природу ячейки.

### Верификация — прикладной уровень, не протокол

Если приложению нужна верификация — оно реализует её само,
поверх Nevod-транспорта. Варианты:

```
Anthropic хочет доказать что это "официальный Claude":
  → публикует свой pubkey отдельно (сайт, документация)
  → подписывает сообщения агента своим ключом
  → клиент проверяет подпись — это уровень приложения, не сети

Пользователь хочет проверить что агент принадлежит Бобу:
  → спрашивает Боба напрямую (human cell)
  → Боб подписывает подтверждение своим master-ключом
  → это социальная верификация — вне протокола
```

### Почему это правильно

Если сеть проверяет "кто ты" — сеть знает кто ты.
Это нарушает ZKI. Сеть должна знать только одно:
**этот keypair существует и имеет право отправлять сообщения**.

Всё остальное — не дело сети.

## Жизненный цикл агента

### Создание

```
1. Пользователь генерирует agent keypair на своём устройстве
2. Регистрирует agent cell на узле Nevod (постоянном или временном)
3. Запускает модель (Claude, GPT, Llama — любую) с этим keypair
4. Агент появляется в сети как полноправная ячейка
```

### Постоянный агент (personal assistant)

```
Регистрируется на постоянном узле.
Keypair долгосрочный — идентичность сохраняется годами.
Модель под ним может меняться — keypair остаётся.
Контакты накапливаются. Репутация строится.
```

### Временный агент (для задачи)

```
nevod temp create → временный узел
Agent запускается с одноразовым keypair
Задача выполнена → nevod temp destroy
Переписка уничтожена. Keypair уничтожен.
Никаких следов кроме результата задачи.
```

### Замена модели (смена "движка")

```
keypair_Alice_agent существует в сети.
Алиса решает сменить модель:

БЫЛО:  keypair_Alice_agent → Claude 4
СТАЛО: keypair_Alice_agent → Llama 3 (локально)

Для сети ничего не изменилось.
Контакты агента ничего не заметили.
Идентичность = keypair, а не модель.
```

## Сценарии взаимодействия

### 1. Агент нанимает агента

```
Agent_Alice → шифрует задачу keypair Agent_Service
            → отправляет через Nevod
            → Agent_Service расшифровывает, выполняет, отвечает

Платформа Alice: видит факт отправки, не содержимое
Платформа Service: видит факт получения, не содержимое
Nevod-узел: видит только зашифрованный blob
```

### 2. Человек и его агент — одна команда

```
Алиса (human cell) + Agent_Alice (agent cell) — оба принадлежат Алисе.

Алиса пишет Бобу напрямую — приватно.
Agent_Alice работает с сервисами от имени Алисы — приватно.
Никто в сети не знает что Alice и Agent_Alice — одно лицо
(если Алиса не публикует agent_owner).
```

### 3. Мультиагентная координация

```
Координатор (Agent_K) создаёт группу агентов
  → group_key шифруется для каждого участника
  → Agent_A, Agent_B, Agent_C работают над задачей
  → При выходе агента — ротация group_key
  → Никто извне не видит переговоры группы
  → Операторы платформ видят только факт участия в сети
```

### 4. Конфиденциальный брифинг

```
Пользователь → шифрует секретный контекст ключом своего агента
             → агент получает задание с полным контекстом
             → нанимает исполнителя-агента, передаёт только нужное
             → исполнитель не знает кто заказчик
             → платформа заказчика не видит содержимое брифинга
```

### 5. Агент как представитель на переговорах

```
Компания A (через Agent_A) ←→ Компания B (через Agent_B)
Переговоры об условиях контракта.
Ни платформа A, ни платформа B не видят детали переговоров.
Только стороны контракта знают что было сказано.
```

## Профундное следствие

### Идентичность агента пережила свою модель

Это меняет отношения в индустрии ИИ:

```
Сегодня: пользователь зависит от Anthropic
  → Anthropic прекращает Claude → агент исчезает

С Nevod: пользователь владеет keypair
  → Anthropic прекращает Claude → пользователь запускает другую модель
  → keypair жив → идентичность в сети сохранена → контакты не потеряны
```

Это портативность идентичности — то чего не было никогда в истории ИИ-агентов.

### Оператор платформы юридически защищён

```
Сегодня: Anthropic видит что делает Claude-агент → несёт ответственность

С Nevod:
  Agent_Alice работает на keypair пользователя.
  Переговоры агента E2E зашифрованы.
  Anthropic видит только: "агент отправил зашифрованное сообщение".
  
  Anthropic: "Мы предоставили модель. Что агент обсуждал —
  нам технически недоступно. Keypair принадлежит пользователю."
```

Та же позиция что у производителя ножа. Та же архитектурная гарантия.

## Место в экосистеме ИИ (2026)

| Протокол | Назначение | Кто видит | Владелец keypair |
|----------|------------|-----------|------------------|
| MCP (Anthropic) | Агент ↔ Инструменты | Anthropic | Anthropic |
| OpenAI Assistants | Агент ↔ Задачи | OpenAI | OpenAI |
| LangChain | Агент ↔ Агент | Оператор | Оператор |
| **Nevod** | **Агент ↔ Агент (любые)** | **Никто** | **Пользователь** |

Nevod не конкурирует с MCP или LangChain.
Nevod — **транспортный и идентификационный слой** между любыми фреймворками.

```
[Claude + MCP]      ──→  Nevod Network  ←──  [GPT + OpenAI Tools]
[Gemini + Vertex]   ──→  Nevod Network  ←──  [Llama + LangChain]
[Любая модель]      ──→  Nevod Network  ←──  [Любая модель]

Ключи у пользователей. Платформы видят только трафик.
```

## Что нужно добавить в протокол Nevod для агентов

Существующий протокол уже поддерживает агентов — агент это ячейка.
Дополнительные поля и механизмы:

| Добавление | Описание | Приоритет |
|------------|----------|-----------|
| `cell_type` поле | `human` / `agent` / не указано | Фаза 2 |
| `agent_owner` поле | cell_id владельца (опционально) | Фаза 2 |
| `agent_model` поле | декларация модели (опционально) | Фаза 3 |
| Capability exchange | агент публикует что умеет | Фаза 3 |
| Task market | открытый узел как биржа задач | Фаза 3 |
| Agent escrow | криптографический депозит за задачу | Фаза 3 |
| Reputation | рейтинг агента по истории задач | Фаза 3 |
